Choice of clustering model¶

As a part of the model training process, positive examples will be clustered in order to understand whether there are distinct sets of bioactive compound "classes". These clusters will be modeled separately and the results joined at the end so as to not create conflicting signals in the model training process.

Choice of clustering algorithm: there is a good (though biased) discussion of potential clustering algorithms here, which steps through some of the main characteristics that are desirable in an algorithm and compares pros/cons.

We do not necessarily have a good idea a priori as to what properties our fingerprint data have or what shapes the compound sets are likely to form in high-dimensional space. However we do know a few things:

  • Though we may be able to perform checks using prior knowledge on input drug sets, we don't want to assume for this step that we know the number of clusters, so any method we choose should have some ability to "help" with this decision. On a related note, we should be able to extract some sort of metric as to whether there are multiple clusters at all.
  • "Don't be wrong": for this use case, it may be better to avoid classifying a given molecule than to allow it to belong to the nearest class. It could create unwanted noise for that class's model, and its information can be incorporated in other ways (e.g. using a Tanimoto approach).
  • The feature space will typically be at least the size of an FP2 fingerprint (1024 bits), so speed could be a limiting factor.

Given the third point, we can remove slow methods from our search altogether. The HDBSCAN fellows (of the post linked above) have conveniently benchmarked the performance of the same set of Python clustering implementations here. At least in terms of scaling with the number of data points (note: not our limitation here), only three methods do reasonably well: K-means variants, agglomerative clustering variants, and DBSCAN variants.

Unapologetically stealing their code and modifying it somewhat to compare feature sizes rather than data point numbers, we have the following clustering benchmark algorithm, which generates and uses toy datasets with Gaussian clusters:

import numpy as np
import pandas as pd
import sklearn.cluster
import sklearn.datasets
import fastcluster
import hdbscan
import time


def benchmark_algorithm(dataset_dimensions, cluster_function, function_args, function_kwds,
                        dataset_size=500, dataset_n_clusters=2, max_time=10, sample_size=2):

    # Initialize the result with NaNs so that any unfilled entries
    # will be considered NULL when we convert to a pandas dataframe at the end
    result = np.nan * np.ones((len(dataset_dimensions), sample_size))
    for index, dataset_dimension in enumerate(dataset_dimensions):
        for s in range(sample_size):
            # Use sklearns make_blobs to generate a random dataset with specified size
            # dimension and number of clusters
            data, labels = sklearn.datasets.make_blobs(n_samples=dataset_size,
                                                       n_features=dataset_dimension,
                                                       centers=dataset_n_clusters)

            # Start the clustering with a timer
            start_time = time.time()
            cluster_function(data, *function_args, **function_kwds)
            time_taken = time.time() - start_time

            # If we are taking more than max_time then abort -- we don't
            # want to spend excessive time on slow algorithms
            if time_taken > max_time:
                result[index, s] = time_taken
                return pd.DataFrame(np.vstack([dataset_dimension.repeat(sample_size),
                                               result.flatten()]).T, columns=['x','y'])
            else:
                result[index, s] = time_taken

    # Return the result as a dataframe for easier handling with seaborn afterwards
    return pd.DataFrame(np.vstack([dataset_dimensions.repeat(sample_size),
                                   result.flatten()]).T, columns=['x','y'])

from sklearn.decomposition import PCA

def plot_pca(dataset, labels=None, return_coords=False):
    pca_model = PCA(n_components=2).fit(dataset)
    pca_coords = pca_model.transform(dataset)
    if return_coords:
        return pca_coords
    fig = plt.scatter(x=pca_coords[:,0], y=pca_coords[:,1], c=labels)
    plt.xlabel(f"PC1 ({round(pca_model.explained_variance_ratio_[0] * 100, 1)}% variance explained)")
    plt.ylabel(f"PC2 ({round(pca_model.explained_variance_ratio_[1] * 100, 1)}% variance explained)")
    return fig

We notice some important points:

  • The size parameter (at least for DBSCAN) matters! At a basic default epsilon value of 0.5, the algorithm doesn't find multiple clusters, despite the obvious modularity (both visually and by design of this dataset). Continuing to increase epsilon by orders of magnitude finds a range in which it can distinguish clusters, then they are lost once again.
  • Points that are the same color can actually be a set that has not been clustered (represented in this implementation as cluster label = -1). This fits with our intuition:
    • When epsilon is too small, no point will be "close" enough to any other point to convince the algorithm that they are from the same cluster, so all points are left unclustered.
    • When epsilon is too big, any variation on a realistic scale will be dwarfed and all points will be dumped into the same cluster (but a "real" cluster with label >=0).
  • Even given a reasonable size parameter, DBSCAN has difficulty with clusters at different "scales". At eps=20, the two dense clusters are found, but the more diffuse set is left unclustered. At eps=200, the algorithm doesn't find the more subtle difference across PC2. HDBSCAN (hierarchical DBSCAN) was developed for this purpose and may be worth looking into.

All of this said, this is a highly contrived example, and is probably atypical given the huge discrepancy between cluster "scales" (separation across PC1 which explains almost 90% of the variance vs. separation across PC2 which explains less than 0.5% of the variance). So, it may be more helpful to check out performance on real data.

A quick note here on visualization of these clusters: the plots in this notebook have all been generated using PCA. Although other methods exist (e.g. t-SNE) that may be more effective in some applications, they often sacrifice "long-range" distance fidelity (do very different points stay farther apart) for finer-grained distinctions (do somewhat different points stay farther apart). Here, we do care about maintaining a conception of large- vs. small-scale substructure, since small-scale substructure is likely less important.

Next, we assess Silhouette scores (metric of modularity used in clustering, range is from -1 to 1, higher is better) across a range of DBSCAN epsilon values to find the putatively optimal epsilon. For some values of epsilon, a non-useful clustering will be returned (all points unclustered or only one "real" cluster) -- we won't include scores for these epsilon levels in the plots.

def get_ss(eps, data):
    unique_data = np.unique(data, axis=0)
    cluster_fit = DBSCAN(eps=eps).fit(unique_data)
    if len(np.unique(cluster_fit.labels_)) == 1:  # Only a single cluster
        return None
    return ss(unique_data, cluster_fit.labels_)

The crux of this notebook¶

It's hard to tell from visual inspection whether they are likely biological relevant or just an artifact of "how zoomed in" we are in this plot. It may be that these "clusters" are tight chemical subspaces explored by different drug companies based on their IP, while all of them share the same basic mode of bioactivity. In this case, it is certainly to our advantage to model them all together.

Ideally, we can find a value of epsilon that is generally consistent in resolving biologically relevant subsets. Finding this value will require guidance using prior biological knowledge.

Conclusions and outstanding questions¶

  • DBSCAN has several advantages that are helpful in our use case, and the implementation in scikit-learn performs at quite a reasonable speed.
  • It seems that effective epsilon (size parameter) values for DBSCAN stay in a reasonable range for our consistent expected feature set (a 1024-bit vector). The relevant range is approximately 6-12 units.
  • Clustering of different drug sets did not successfully find robust clusters, at least for the four common drug targets tested here. Visual analysis of principal component plots did not show complete separation between clusters even after parameter optimization, and silhouette scores were quite modest (optimal values 0.2-0.35, with a theoretical maximum of 1). This could suggest a problem with the clustering method, but it is quite possible that there simply is not a clear set of underlying clusters in these datasets.
  • For our application, we would not like to separate compounds into different models unless there is such a meaningful difference between clusters (biologically and computationally) that models combining their information would be less effective. So, one approach would be to create a silhouette score threshold under which we don't use the clustering. As an example:
    • Run DBSCAN at eps = {0.1,6,8,10,12,100} (0.1 and 100 to ensure the we are not missing something important at a highly different scale).
    • Calculate silhouette scores for each of these clusterings.
    • If the maximum silhouette value applies to either 0.1 or 100, flag the result because something may be wrong.
    • If the maximum silhouette value is below 0.5, leave the compounds in a single cluster.
    • Otherwise, pass the discovered clusters on to model training.
  • Would it be better to choose a rigid epsilon that is consistent across different drugs? This value could be set to be high (more "conservative") such that we avoid unwanted subclusterings that only provide minor benefit.
  • It would be helpful to find an example of a drug target known to have several unique classes of inhibitors as a positive control (to understand what epsilon value achieves proper clustering for that example and what silhouette score such a clustering returns).
  • Methods like DBSCAN and HDBSCAN can and do leave data points unclustered if they don't fit well into the current set of clusters. Would we rather use an algorithm like this to remove potential noise and not include those "outliers" in training, or do we think they contain important signal? If we include them, do we force them into the nearest cluster, model the set of "misfits" as an independent group, or use high-stringency Tanimoto comparisons to these misfits to complement the model-based hits from more robust clusters?
